15 /**
16  * \ingroup fileio
17  *
18  * The block cache implements a simple key-value store for extremely large
19  * values (~16MB at least). Every key can only be written to exactly once,
20  * and allows for arbitrary range reads (i.e. read byte X to byte Y of this key)
21  *
22  * Essentially every value is stored as a single file inside the
23  * storage_prefix parameter set at \ref init.
24  *
25  * The block_cache is safe for concurrent use.
26  *
27  * Use On a Distributed File System
28  * --------------------------------
29  *
30  * The storage prefix be located on a distributed filesystem
31  * (for instance HDFS or NFS). In which case, *every* machine sharing the same
32  * storage prefix also shares keys.
33  *
34  * When sharing a storage prefix with other processes on a distributed
35  * filesystem, the atomicity guarantees of the filesystem becomes important.
36  *
37  * In particular, on HDFS, you may find keys in a "indeterminate" state, where
38  * it cannot be written to, but cannot be queried (because the writer has
39  * created the file but has not finished writing to it yet). On NFS multiple
40  * machines may be able to write to the same key, but only one will win. Also
41  * the length and contents of the key may be wrong if you read the key
42  * while someone else is writing to it.
43  *
44  * Design Notes
45  * ------------
46  * We will like these "interesting" distributed file system properties to not
47  * be true when the block_cache is merely used concurrently. So a bit of care
48  * is needed to ensure atomicity, at least within the context of the same
49  * block_cache object. Essentially we want write-once, but arbitrary parallel
50  * reads semantics.
51  */
